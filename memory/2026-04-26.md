## memtensor 插件修复（2026-04-26 16:00-17:00）

### 问题描述
用户已完成基础安装（npm包/symlink/config），但 memtensor daemon 无法启动，导致：
- `is_daemon_running()` → False
- `discover_memory_providers()` → memtensor: available=False
- 启动时报 `Cannot find module 'C:\mnt\c\Users\pc\.hermes\memos-plugin\node_modules\tsx\dist\cli.mjs'`

### 根因分析（4个bug叠加）

**Bug 1: bridge_path.txt 路径格式错误**
- 内容：`/c/Users/pc/.hermes/memos-plugin/bridge.cts`（MINGW64格式）
- 导致 Path() 解析为 `\\c\\Users\\...`（不存在路径）
- 修复：写入正确 Windows 路径 `C:/Users/pc/.hermes/memos-plugin/bridge.cts`

**Bug 2: _resolve_tsx() 返回 node.exe 而非 tsx**
- 原代码硬编码返回 `C:/Program Files/nodejs/node.exe`
- node.exe 不知道如何执行 `.cts` TypeScript 文件
- 需要 tsx 来解释执行 TypeScript
- 修复：返回 `~/.hermes/memos-plugin/node_modules/.bin/tsx.cmd`

**Bug 3: find_bridge_script() 路径拼接错误**
- `_get_plugin_root()` → `plugins/memory/memtensor/../..` → 错误地指向 `hermes-agent/plugins/`
- 导致 `plugin_root / "bridge.cts"` 找不到正确文件
- 修复：通过正确的 `bridge_path.txt` 绕开这个错误

**Bug 4: daemon_manager.py Windows 不兼容**
- `select.select()` 在 Windows 上对 subprocess stdout pipe 报 `OSError [WinError 10038]`
- `open(cfg_path).read_text()` 在 GBK Windows 上导致 `UnicodeDecodeError`
- 修复：
  - 将 `select.select` 替换为 threading + timeout 的轮询
  - 添加 `encoding="utf-8", errors="replace"`

### 修复文件清单
| 文件 | 修改内容 |
|------|---------|
| `hermes-agent/plugins/memory/memtensor/bridge_path.txt` | MINGW64路径 → Windows路径 |
| `hermes-agent/plugins/memory/memtensor/config.py` | `_resolve_tsx()` + JSON读取encoding |
| `hermes-agent/plugins/memory/memtensor/daemon_manager.py` | select.select替换 + 读文件encoding |

### 验证结果
```
✅ is_daemon_running() → True
✅ discover_memory_providers() → memtensor: available=True
✅ MemosCoreBridge().ping() → {'pong': True}
✅ MemosCoreBridge().ingest() → 成功
✅ MemosCoreBridge().search() → 命中记忆内容
```

### 教训
1. MINGW64 路径格式（`/c/...`）不能直接给 Windows Python Path 使用
2. `_resolve_tsx()` 硬编码 `node.exe` 是错误的——tsx 是 TypeScript 解释器，不是 node 别名
3. Windows subprocess 的 stdout pipe 不能用 `select.select()`——只能用 threading 轮询或 `communicate()`

## 2026-04-26 22:00 — claude-mem 记忆插件启动失败修复

### claude-mem 定位
- **claude-mem** = Claude Code 的持久化记忆系统（插件），不是 AI
- 端口：37777，记忆视图：http://localhost:37777
- 为 Claude Code 提供跨会话持久化记忆

### 问题
- claude-mem 无法启动，端口 37777 无响应
- `health: {"status":"ok", "activeSessions":14}` 但服务实际未运行

### 根因
- `worker.pid` 文件存的是**重启前的旧 PID**
- 该 PID（例：12384）已被 Windows `TextInputHost.exe` 回收复用
- claude-mem 检测到 PID 存在，误以为已有实例在跑，**跳过真正的启动**

### 修复
- 删除 stale `worker.pid` 文件
- 重新拉起 claude-mem
- 结果：✅ PID 10716，Port 37777，health ok

### 教训
- **Windows PID 回收复用**：进程退出后 PID 可能被系统进程（TextInputHost.exe）占用
- 依赖 PID 文件做单实例锁时，不能只检查 PID 是否存在，必须同时验证进程命令行匹配
